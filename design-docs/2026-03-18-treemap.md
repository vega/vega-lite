# Treemap Support in Vega-Lite: Design Document

## Summary

We propose adding treemap visualization support to Vega-Lite. This introduces two new pieces of syntax:

1. **`"mark": "treemap"`** — a new first-class mark (like `geoshape`). Under the hood, it compiles to a Vega `rect` mark and automatically injects the Vega `treemap` layout transform to compute rectangle coordinates. The user never writes the layout transform directly. (See [Decision: Treemap as mark](#treemap-as-mark-preferred-vs-exposing-treemap-transform).)

2. **`"hierarchy"` encoding channel** — a new compound encoding channel that specifies how flat data should be converted into a tree. The treemap mark reads this channel and automatically injects the appropriate Vega hierarchy transform (`stratify` or `nest`). (See [Decision: Hierarchy as encoding channel](#hierarchy-as-encoding-channel-preferred-vs-separate-transforms).)

## Data models

Treemaps require hierarchical (tree-structured) data. We propose supporting two flat data models via the `hierarchy` encoding channel. Pre-nested JSON is discussed as future work.

### Flat data with parent references

The most common tabular format (CSV, databases). Each row has an ID and a parent ID. The `hierarchy` encoding channel specifies which fields contain these.

**Proposed Vega-Lite spec:**

```json
{
  "data": {
    "values": [
      {"id": 1, "parent": null, "name": "Root", "size": 0},
      {"id": 2, "parent": 1, "name": "A", "size": 100},
      {"id": 3, "parent": 1, "name": "B", "size": 200},
      {"id": 4, "parent": 2, "name": "A1", "size": 50},
      "..."
    ]
  },
  "mark": "treemap",
  "encoding": {
    "hierarchy": {
      "key": {"field": "id"},
      "parentKey": {"field": "parent"}
    },
    "size": {"field": "size", "type": "quantitative"},
    "color": {"field": "name", "type": "nominal"}
  }
}
```

**Compiled Vega output** — the `hierarchy` encoding injects `stratify`, then the treemap mark injects the layout transform:

```json
{
  "data": [
    {
      "name": "source",
      "values": [
        {"id": 1, "parent": null, "name": "Root", "size": 0},
        {"id": 2, "parent": 1, "name": "A", "size": 100},
        "..."
      ],
      "transform": [
        {"type": "stratify", "key": "id", "parentKey": "parent"},
        {
          "type": "treemap",
          "field": "size",
          "method": "squarify", // default; customizable via mark def
          "size": [{"signal": "width"}, {"signal": "height"}]
        },
        {"type": "filter", "expr": "!datum.children"} // default nodes: "leaves"
      ]
    }
  ],
  "marks": [
    {
      "type": "rect",
      "from": {"data": "source"},
      "encode": {
        "update": {
          "x": {"field": "x0"},
          "x2": {"field": "x1"},
          "y": {"field": "y0"},
          "y2": {"field": "y1"},
          "fill": {"scale": "color", "field": "name"}
        }
      }
    }
  ]
}
```

The `size` encoding is optional — if omitted, all leaf nodes are weighted equally (the layout uses node count). This follows `geoshape`'s precedent where `shape` is not required.

The `method`, `padding`, `ratio`, and other layout parameters come from the mark def (e.g. `"mark": {"type": "treemap", "method": "binary"}`). See [Mark definition properties](#mark-definition-properties) for the full list.

### Flat data with categorical grouping

No explicit parent-child links in the data. The hierarchy is derived by grouping on categorical fields. The `hierarchy` encoding channel takes an object with a `nest` key containing an array of field defs — the grouping order goes from root to leaf.

**Proposed Vega-Lite spec:**

```json
{
  "data": {
    "values": [
      {"department": "Engineering", "region": "East", "name": "Alice", "salary": 120},
      {"department": "Engineering", "region": "West", "name": "Bob", "salary": 110},
      {"department": "Sales", "region": "East", "name": "Carol", "salary": 90},
      {"department": "Sales", "region": "West", "name": "Dave", "salary": 95},
      "..."
    ]
  },
  "mark": "treemap",
  "encoding": {
    "hierarchy": {"nest": [{"field": "department"}, {"field": "region"}]},
    "size": {"field": "salary", "type": "quantitative", "aggregate": "sum"},
    "color": {"field": "department", "type": "nominal"}
  }
}
```

**Compiled Vega output** — same pattern but with `nest` instead of `stratify`:

```json
{
  "data": [
    {
      "name": "source",
      "values": ["..."],
      "transform": [
        {"type": "nest", "keys": ["department", "region"]},
        {
          "type": "treemap",
          "field": "salary",
          "method": "squarify",
          "size": [{"signal": "width"}, {"signal": "height"}]
        }
      ]
    }
  ],
  "marks": [{"type": "rect", "...": "same as above"}]
}
```

### Pre-nested JSON (future work)

Pre-nested JSON — where the tree structure is expressed directly with `children` arrays — is a valid tree format but is not supported in the initial implementation.

```json
{
  "data": {
    "values": [
      {
        "name": "Root",
        "children": [
          {"name": "A", "size": 100, "children": [{"name": "A1", "size": 50}]},
          {"name": "B", "size": 200}
        ]
      }
    ]
  },
  "mark": "treemap",
  "encoding": {
    "size": {"field": "size", "type": "quantitative"},
    "color": {"field": "name", "type": "nominal"}
  }
}
```

Vega's `treemap` layout transform requires an internal `.root` property on the data source — a D3 hierarchy node set by `stratify` or `nest` during dataflow execution. It also requires all tree nodes to be present as individual tuples in the flat data source array (so the `rect` mark can iterate over them). Pre-nested JSON satisfies neither requirement: the data source has a single root object, and child nodes are nested inside it rather than in the array.

Supporting pre-nested JSON would require a new Vega transform that:

1. Flattens the nested tree into individual tuples in the data source array
2. Calls `d3.hierarchy()` to build the internal tree structure
3. Sets `.root` on the data source

This is feasible but requires new Vega-level infrastructure. In a future milestone, this could be exposed as a third `hierarchy` encoding subtype, alongside the existing stratify and nest object shapes.

## How it works under the hood

### The `treemap` mark

The proposed `treemap` mark would compile to a Vega `rect` mark. It automatically injects three Vega transforms into the data pipeline:

1. **Hierarchy transform** — `stratify` or `nest`, determined by the shape of the `hierarchy` encoding channel. This converts flat data into a tree and sets the internal `.root` property that Vega's layout transforms require.
2. **Layout transform** — Vega's `treemap` transform, which reads `.root` and computes `x0`, `y0`, `x1`, `y1` coordinates for each node's rectangle, sized to `[{signal: "width"}, {signal: "height"}]`.
3. **Node filter** — a Vega `filter` transform, controlled by the `nodes` mark property (default `"leaves"`). This filters to the requested node type: `"leaves"` emits `{"type": "filter", "expr": "!datum.children"}`, `"internal"` emits `{"type": "filter", "expr": "datum.children"}`, and `"all"` skips the filter. (See [Decision: Node filtering](#node-filtering-nodes-mark-property).)

The mark compiler maps the layout output to Vega rect encoding: `x0` → `x`, `x1` → `x2`, `y0` → `y`, `y1` → `y2`. The `x` and `y` encoding channels are blocked for treemap since positions come from the layout. The user's `size` encoding tells the layout which field determines rectangle area; if `size` is omitted, the layout defaults to node count (equal-weight rectangles). The `size` channel does not generate a visual scale for treemap — the layout transform handles sizing internally (similar to how `geoshape` skips the `shape` scale). Standard channels like `color`, `tooltip`, `opacity` pass through to the rect as usual.

The user never writes any of these Vega transforms — all are internal implementation details, just like `geoshape` automatically injects Vega's `geoshape` transform for projections. (See [Decision: Treemap as mark](#treemap-as-mark-preferred-vs-exposing-treemap-transform).)

### The `hierarchy` encoding channel

The `hierarchy` channel is a compound encoding channel whose shape determines which Vega hierarchy transform to inject:

- **Object with `key` + `parentKey`** → injects Vega `stratify` transform
- **Object with `nest`** → injects Vega `nest` transform

Both modes are objects discriminated by their required keys (`key`+`parentKey` vs `nest`), matching Vega-Lite's dominant pattern for mode discrimination (see [Decision: Nest syntax](#nest-syntax-nest-wrapper-preferred-vs-bare-array)).

This is a new pattern for VL — existing encoding channels map a single field to a visual property. The `hierarchy` channel maps one or more fields to a _structural_ property (the tree). However, VL already has non-visual encoding channels (`detail` for grouping, `order` for sorting, `key` for object constancy), so structural channels have precedent.

Unlike visual channels, `hierarchy` does not generate scales or legends — it is consumed during data parsing to inject transforms. Structural validation (e.g. stratify mode requires both `key` and `parentKey`; nest mode requires `nest`) is enforced by TypeScript types and the JSON schema, the same way other encoding channels validate their shape. Sub-fields use the standard `{field: ...}` syntax for consistency with other encoding channels.

(See [Decision: Hierarchy as encoding channel](#hierarchy-as-encoding-channel-preferred-vs-separate-transforms).)

### Mark definition properties

These layout parameters would go on the mark def (e.g. `"mark": {"type": "treemap", "method": "binary"}`). They control the treemap layout algorithm, not data field mappings. All defaults match Vega's defaults.

| Property | Default | Description |
| --- | --- | --- |
| `method` | `"squarify"` | Layout algorithm: `squarify`, `resquarify`, `binary`, `dice`, `slice`, `slicedice` |
| `padding` | `0` | Sets both paddingInner and paddingOuter |
| `paddingInner` | `0` | Space between sibling nodes |
| `paddingOuter` | `0` | Space between parent boundary and children |
| `paddingTop` / `paddingRight` / `paddingBottom` / `paddingLeft` | — | Directional outer padding |
| `ratio` | `~1.618` (golden ratio) | Target aspect ratio for squarify/resquarify |
| `round` | `false` | Round coordinates to integers |
| `nodes` | `"leaves"` | Which tree nodes to render: `"leaves"`, `"internal"`, or `"all"` |

### Future work: Labels

The treemap mark produces rectangles only. Labels would require the treemap mark to emit both a Vega `rect` and a Vega `text` mark from the same layout pipeline, since a separate text mark in a VL layer would not have access to the internally-computed layout coordinates.

Treemap labels in practice need data-driven styling — for example, [Vega's treemap example](https://vega.github.io/vega/examples/treemap/) scales `fontSize` and `fillOpacity` by `depth` (a field the Vega treemap transform outputs alongside `x0`/`y0`/`x1`/`y1`). This points toward a `label` encoding with nested channels:

```json
{
  "encoding": {
    "label": {
      "text": {"field": "name", "type": "nominal"},
      "fontSize": {"field": "depth", "type": "quantitative", "scale": {"range": [14, 28]}},
      "opacity": {"field": "depth", "type": "quantitative", "scale": {"range": [0.15, 1.0]}}
    }
  }
}
```

However, nested encoding is a new pattern for Vega-Lite that would need new infrastructure for validation, scale resolution, and compilation. This is not specific to treemaps — other marks (e.g., bar charts, pie charts, geographic marks) would also benefit from a `label` encoding channel. This should be designed holistically as a cross-cutting feature rather than a treemap-specific addition.

## Decisions

### Treemap as mark (preferred) vs exposing treemap transform

**Proposed:** Expose treemap as `"mark": "treemap"` — a high-level mark that internally handles all Vega transforms (hierarchy construction + layout). The user never writes Vega transforms directly.

Vega-Lite is fundamentally a macro language over Vega. Exposing Vega's `treemap` and `stratify` transforms directly would be too low-level for VL's abstraction — users would need to understand the two-step pipeline (build hierarchy, then compute layout), wire coordinate fields (`x0`/`y0`/`x1`/`y1`) to a `rect` mark manually, and set `size` to `[{signal: "width"}, {signal: "height"}]`. VL already has precedent for hiding this kind of plumbing:

- **`geoshape`** — automatically injects Vega's `geoshape` transform and `geopoint`/`geojson` data nodes. The user writes `"mark": "geoshape"` + `"projection": {...}` and never touches the underlying transforms.
- **`bar`/`area` with stacking** — when the user encodes `color` on a `bar` mark, VL automatically injects the Vega `stack` transform to compute `y0`/`y1` stacked coordinates. The user never writes `{"type": "stack", ...}`.

The treemap mark would follow the same principle: both the Vega `stratify`/`nest` and `treemap` transforms are internal implementation details.

#### Sub-decision: First-class mark (preferred) vs composite mark

Given that treemap is a mark, we propose implementing it as a **first-class mark** (with its own mark compiler, like `geoshape`) rather than a **composite mark** (like `boxplot`).

| Aspect              | `geoshape` (existing)             | `treemap` (proposed)                    |
| ------------------- | --------------------------------- | --------------------------------------- |
| Mark enum           | `Mark.geoshape`                   | `Mark.treemap`                          |
| Vega mark type      | `shape`                           | `rect`                                  |
| Internal transforms | `geoshape`, `geopoint`, `geojson` | `stratify`/`nest` + `treemap`           |
| Config channel      | `projection: {type: "albersUsa"}` | `hierarchy: {key: ..., parentKey: ...}` |
| Layout params       | `projection` properties           | `mark: {method: "squarify", ...}`       |

**Rejected alternative: Composite mark (like `boxplot`).** A composite mark normalizer would expand `mark: "treemap"` into `mark: "rect"` + transforms during normalization. This is not inherently wrong, and users could even layer their own `text` mark for labels since both sub-marks would share the same data source. However, this was rejected because:

- **Technical constraints (primary).** Normalization runs before model construction, so there is no direct access to width/height signals. The treemap layout transform requires `[{signal: "width"}, {signal: "height"}]` for sizing — these are only available after model construction, which a first-class mark compiler has access to but a composite mark normalizer does not.
- **Shared encoding namespace problem (forward-looking).** Composite marks share one encoding across all sub-layers, but different sub-marks interpret the same channels differently. VL's line + point overlay suffers from this: `size` encoding applies to both point radius and line width ([#6841](https://github.com/vega/vega-lite/issues/6841)). For treemap labels (future work), a composite mark would apply `color` to both the rect fill and the label text color — but labels typically need a contrasting color (e.g., white text on dark rectangles). Similarly, `opacity` would apply identically to both rect and text, when often you want full-opacity labels on semi-transparent rectangles. A first-class mark compiler can route each channel to the appropriate sub-mark independently.

### Hierarchy as encoding channel (preferred) vs separate transforms

**Proposed:** Add `hierarchy` as a compound encoding channel on the treemap mark, rather than exposing `stratify`/`nest` as separate user-facing transforms.

Vega's `treemap` layout transform always requires a hierarchy to be built first (via `stratify` or `nest`), which sets an internal `.root` property on the data source. The treemap mark cannot function without this step. Bundling hierarchy construction into the mark (via the `hierarchy` encoding channel) keeps the spec concise and avoids exposing Vega-internal plumbing to users.

The channel's shape discriminates the mode:

- Object `{key: ..., parentKey: ...}` → inject Vega `stratify`
- Object `{nest: [{field: ...}, ...]}` → inject Vega `nest`

Both modes are objects with distinct required keys, matching Vega-Lite's dominant discrimination pattern (see [Decision: Nest syntax](#nest-syntax-nest-wrapper-preferred-vs-bare-array)).

**Rejected alternative: Separate `stratify`/`nest` user-facing transforms.** This was the original design, which assumed the treemap mark could work standalone with pre-nested JSON (no upstream transform needed). Investigation revealed this is not the case — Vega's treemap transform always requires `.root` to be set by a prior `stratify` or `nest` transform. Since hierarchy construction is always required, exposing it as a separate transform adds verbosity without benefit. Bundling it into the `hierarchy` encoding channel is more user-friendly and follows VL's principle of inferring transforms from encoding (like `bar` + `color` → stack).

### Node filtering: `nodes` mark property

**Problem:** Vega's [treemap example](https://vega.github.io/vega/examples/treemap/) renders internal nodes and leaf nodes as separate layers with different visual properties — internal nodes get category-colored fills (backgrounds), leaf nodes get transparent fill with white strokes (grid lines). A single flat layer of identically styled rectangles cannot reproduce this canonical look because the two node types need independent encodings (`color`, `fill`, `stroke`, `opacity`).

**Decision:** Add a `nodes` property to the treemap mark definition that controls which tree nodes are rendered. The filter is applied after the implicit treemap layout transform (where computed fields like `children` and `depth` become available).

```json
"mark": {"type": "treemap", "nodes": "leaves"}
```

Values: `"leaves"` (default) | `"internal"` | `"all"`

- `"leaves"` — only nodes without children (the most common treemap rendering)
- `"internal"` — only nodes with children (root + intermediate nodes)
- `"all"` — every node in the tree

The default of `"leaves"` produces the standard treemap look out of the box. Users who want the layered Vega-style look compose two treemap marks with VL's existing `layer`, each with independent encodings. Shared channels like `hierarchy` and `size` can be specified once at the layer level:

```json
{
  "data": {"url": "data/flare.json"},
  "encoding": {
    "hierarchy": {"key": {"field": "id"}, "parentKey": {"field": "parent"}},
    "size": {"field": "size", "type": "quantitative"}
  },
  "layer": [
    {
      "mark": {"type": "treemap", "nodes": "internal"},
      "encoding": {
        "color": {"field": "name", "type": "nominal"}
      }
    },
    {
      "mark": {"type": "treemap", "nodes": "leaves", "fill": "transparent", "stroke": "#fff"},
      "encoding": {
        "tooltip": [
          {"field": "name", "type": "nominal"},
          {"field": "size", "type": "quantitative"}
        ]
      }
    }
  ]
}
```

**Compiled Vega output:** The `nodes` property emits a Vega `filter` transform after the `treemap` layout transform — `{"type": "filter", "expr": "!datum.children"}` for `"leaves"`, `{"type": "filter", "expr": "datum.children"}` for `"internal"`, or no filter for `"all"`.

In a `layer` spec, VL's dataflow optimizer (`MergeIdenticalNodes` in `src/compile/data/optimizers.ts`) merges identical transform nodes at forks. Since both layers share the same `stratify` + `treemap` pipeline, the shared transforms run once in the compiled output — the graph diverges only at the post-layout `filter` step.

**Rejected alternatives:**

- **Composite mark** — auto-expanding into node/leaf layers would contradict the first-class mark decision and suffers from the shared encoding namespace problem (`color`, `fill`, `stroke`, `opacity` would apply identically to both layers, but the two layers need fundamentally different visual properties).
- **Sub-mark property** (e.g., `"nodes": {"fill": ...}` analogous to `line`'s `"point"` property) — does not allow independent encoding control for the internal node layer, which typically needs entirely different visual properties than the leaf layer.

### Nest syntax: `{nest: [...]}` wrapper (preferred) vs bare array

**Proposed:** The nest mode of the `hierarchy` encoding uses a wrapper object `{"nest": [{"field": "department"}, ...]}` rather than a bare array `[{"field": "department"}, ...]`.

**Rationale:**

1. **Matches VL's dominant discrimination pattern.** When a property has multiple modes, VL uses objects with distinct required keys: `data` (`values` vs `url` vs `name`), `condition` (`test` vs `param`), `lookup.from` (`data` vs `param`), transforms (`filter` vs `calculate` vs ...). Both hierarchy modes are now objects discriminated by their required keys (`key`+`parentKey` vs `nest`). VL has no precedent for using a bare array to mean "switch to a fundamentally different mode" — arrays in encoding (`tooltip`, `detail`, `order`) always mean "multiple of the same thing."

2. **Extensibility.** When pre-nested JSON support is added, the discrimination stays clean — all objects with distinct required keys:
   - `{key: ..., parentKey: ...}` → stratify
   - `{nest: [...]}` → nest
   - `{children: "children"}` → pre-nested JSON (future)

   A bare array would create a mixed array/object/object union, which is harder to discriminate in JSON Schema (`anyOf` with an array type and two object types).

3. **Readability.** The `nest` key makes the mode self-documenting. A bare array of `{field: ...}` inside the `hierarchy` channel is ambiguous — it resembles the "multiple field defs" pattern used by `tooltip`/`detail`, rather than signaling a distinct operating mode.

**Rejected alternative: Bare array `[{field: ...}, ...]`.** Slightly more concise (one fewer nesting level), and `sort` does use bare arrays for explicit domain order. However, `sort`'s bare arrays contain primitive values (strings/numbers), not `{field: ...}` objects — and they represent a list of domain values, not a mode switch. The conciseness gain is marginal (one extra key) and does not justify breaking VL's object-based discrimination pattern.

### Planned extension: depth-based filtering

The Vega `treemap` transform already produces a `depth` field on every node. A natural extension is to allow `nodes` to accept an object form for depth-based filtering, without breaking the existing string API:

```typescript
// Current string values (unchanged)
nodes: 'leaves' | 'internal' | 'all';

// Object form (future)
nodes: {
  maxDepth: 2;
} // → filter: datum.depth <= 2
nodes: {
  depth: [1, 3];
} // → filter: datum.depth >= 1 && datum.depth <= 3
```

This is straightforward to implement since it compiles to the same `filter` transform pattern, just with a `datum.depth` expression instead of `datum.children`.

## Extensibility

### Other tree-based visualizations

The `hierarchy` encoding channel would be reusable across all hierarchy marks — only the mark (and its internal layout transform) changes.

**[Sunburst](https://vega.github.io/vega/examples/sunburst/) (radial partition):**

```json
{
  "mark": "sunburst",
  "encoding": {
    "hierarchy": {"key": {"field": "id"}, "parentKey": {"field": "parent"}},
    "theta": {"field": "size", "type": "quantitative"},
    "color": {"field": "name", "type": "nominal"}
  }
}
```

Under the hood, `sunburst` would inject Vega's `partition` layout transform and compile to a Vega `arc` mark, mapping the computed `x0`/`x1` to start/end angles and `y0`/`y1` to inner/outer radii.

**[Circle packing](https://vega.github.io/vega/examples/circle-packing/) (hierarchical):**

```json
{
  "mark": "circle-pack", // name TBD: circle-pack vs point-pack
  "encoding": {
    "hierarchy": {"key": {"field": "id"}, "parentKey": {"field": "parent"}},
    "size": {"field": "size", "type": "quantitative"},
    "color": {"field": "depth", "type": "quantitative"}
  }
}
```

Under the hood, this mark would inject Vega's `pack` layout transform and compile to a Vega `symbol` mark, mapping the computed `x`, `y`, and `r` to position and size. Note: the name needs care to distinguish from the non-hierarchical [packed bubble chart](https://vega.github.io/vega/examples/packed-bubble-chart/), which uses a `force` layout on flat data rather than the `pack` hierarchy transform.

### Beyond hierarchies: the "mark as layout" pattern

The broader pattern here — a mark that internally injects a layout transform — is not limited to tree visualizations.

**[Word cloud](https://vega.github.io/vega/examples/word-cloud/):**

```json
{
  "data": {
    "values": [{"word": "Vega", "count": 120}, {"word": "Lite", "count": 95}, {"word": "Data", "count": 80}, "..."]
  },
  "mark": "wordcloud",
  "encoding": {
    "text": {"field": "word", "type": "nominal"},
    "size": {"field": "count", "type": "quantitative"}
  }
}
```

Under the hood, `wordcloud` would inject Vega's `wordcloud` transform (which computes `x`, `y`, `fontSize`, `angle` for each word) and compile to a Vega `text` mark. No hierarchy transforms needed — this is flat data with a layout-only mark.

## Implementation

### Phase 1: Treemap mark with stratify and nest hierarchy modes

1. Add `'treemap'` to `Mark` enum in [`src/mark.ts`](src/mark.ts)
2. Add `hierarchy` encoding channel to [`src/channel.ts`](src/channel.ts) and [`src/encoding.ts`](src/encoding.ts), supporting both stratify (`key` + `parentKey`) and nest (`nest` array) modes
3. Create `TreemapNode` in `src/compile/data/treemap.ts` — emits the hierarchy transform (`stratify` or `nest`, determined by the `hierarchy` encoding shape) followed by `{type: "treemap", size: [width, height], ...}` and the node filter
4. Wire into [`src/compile/data/parse.ts`](src/compile/data/parse.ts) — detect treemap mark + hierarchy encoding, insert node
5. Wire into [`src/compile/data/assemble.ts`](src/compile/data/assemble.ts) — add to walker
6. Create mark compiler in `src/compile/mark/treemap.ts` — `vgMark: 'rect'`, `encodeEntry()` maps x/x2/y/y2 from x0/y0/x1/y1
7. Register in [`src/compile/mark/mark.ts`](src/compile/mark/mark.ts) — add to `markCompiler` registry
8. Add treemap to `getSupportedMark` for `SIZE`, `COLOR`, `HIERARCHY` in [`src/channel.ts`](src/channel.ts); block `x` and `y` channels for treemap (positions come from the layout)
9. Add example specs and tests for both stratify and nest modes
10. Update documentation site and example gallery
